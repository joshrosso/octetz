The `for` keyword creates loops. Go's standard library does not include
alternative loop styles such as while, do, or [higher-order function
mapping](https://en.wikipedia.org/wiki/Map_(higher-order_function)). By
exclusively offering `for`, Go codebases have consistency in their iteration
and, as you'll see, can still acomplish what other styles do.

## for

At its most basic level, `for` can be used to create an infinite loop.

```go
for {
	// logic
}
```

Assuming it should end, it is up to you to [break](#continue-and-break) the loop.

## Index based

Iterating over an index in Go looks as follows:

```go
for i := 0; i < 10; i++ {
  fmt.Println(i)
}
```

## Range based (for each)

The `range` keyword can accompany `for` to iterate over lists or maps.
This is simlar to a [foreach loop](https://en.wikipedia.org/wiki/Foreach_loop)
in other languages. Here is an example iterating over a list of names:

```go
names := []string{"sam", "sarah", "parker"}
for i, v := range names {
	fmt.Printf("%d: %s\n", i, v)
}
```

Variable `v`, in the above, is a **copy** of the value at `names[i]`. If you
alter `v` in the loop, the changes will **not** be reflected in the `names`
list: 

```go
names := []string{"sam", "sarah", "parker"}
for _, v := range names {
	if v == "sam" {
		v = "samantha"
	}
}
fmt.Println(names)
```

As expected sam's name is unchanged, outputting:

```txt
[sam sarah parker]
```

_[Go playground example](https://go.dev/play/p/bHsiLFNY3cs)_

To alter the name, you must access it using the index. Additionally, the value
`v` is not needed to satisfy this. Thus, the `for ... range` loop can be written
to only capture `i`:

```go
names := []string{"sam", "sarah", "parker"}
for i := range names {
	if names[i] == "sam" {
		names[i] = "samantha"
	}
}
fmt.Println(names)
```

This results in `name[0]` being reassigned to `samantha`, outputting:

```txt
[samantha sarah parker]
```

## continue and break

`continue` and `break` are common keywords used in loops.

The `continue` keyword stops currerntly executing iteration and moves to the
next. Assume the number 4 is your unlucky number and you do not want it processed or printed:

```go
for i := 0; i < 8; i++ {
	// do some work
	if i == 4 {
		continue
	}
	// do some more work
	fmt.Println(i)
}
```

## Labels

Labels enable naming a loop so that you can `continue` or `break` to that loop.
For example, consider you're processing a static list of orders for your
customers and if an item in the order is out of stock in that customer's region,
you want to skip the customer entirely and move on to the next customer. The
following ugly control-flow describes this:

```go
customer: // label for outermost loop
	// for each customer
	for _, customer := range customers {
		// for each order associated with the customer
		for _, order := range customer.orders {
			// for each item in the order
			for _, item := range order.items {
				// check if item is avaiable in the customer's region
				if !isAvail(item, customer.region) {
					fmt.Printf("[UNAVAILAB]: %s's %s; skipping to next customer\n", customer.name, item)
					continue customer
				}
				fmt.Printf("[PROCESSED]: %s's %s\n", customer.name, item)
			}
		}
	}
}
```

In the above, dave's order included:

* battery 
* charger
* dog
* cat

peggy's order included:

* marker
* staples
* tie

Assuming that `charger`s are unavailable in Dave's region, the above should
process dave's battery then move on to peggy's order:

```txt
[PROCESSED]: dave's battery
[UNAVAILAB]: dave's charger; skipping to next customer
[PROCESSED]: peggy's marker
[PROCESSED]: peggy's staples
[PROCESSED]: peggy's tie
```

[Go playground example](https://go.dev/play/p/d7dLMSKmIFc)

Had we used `break` instead of `continue`, the loop would have broken at the
outer-most loop. This would have ended the looping entirely and never processed peggy's orders.

One could argue that `continue` and `break` alone, while perfectly iditomatic,
can make code harder to read. Labels certainly compound this issue. While labels
are justfieid at times, be sure to consider the simplest way you can express
your iteration needs before adding complexity to the control flow.

## Other styles

Go's absencse of other loop styles like `while` shouldn't limit your abilities. You
just need to consider how to express your needs based on the what was discussed
above. Consider this example where a Java while loop attemps to print a message until a "scheduled" date is reached:

```java
scheduledDate = new Date("01 Apr 2029");
now = new Date().getTime();
while(now < scheduledDate) {
	System.out.println("scheduled date not reached!");
	now = new Date().getTime();
}
```

In Go the same can be represented as a loop that breaks under a specific
condition. The Go-equivelant would be:

```go
t, _ := time.Parse("02 Jan 2006", "01 Apr 2029")
date := t.Unix()
today := time.Now().Unix()

for {
	if date < today {
		break
	}
	
	fmt.Println("scheduled date not reached!")
}
```

This demonstrates Go's commitment for constency by having 1 way to do loops. The trade-off to this is, as you can see, is verbosity.
